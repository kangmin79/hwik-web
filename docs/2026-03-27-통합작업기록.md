## 시작 → 종료
- 검색 정확도: 7% → **90%**
- 손님 매칭: 없음 → **100%** (59건 전부 정확)
- 25개 구별 검색: **100%**, FAIL: **0**

## 주요 작업

### 검색 엔진
- 로컬 파서(localParseQuery): Claude API 없이 즉시 파싱 (거래유형/카테고리/지역/가격/단지명)
- SQL 직접 검색: 벡터 검색은 보조, 구조화된 조건은 SQL WHERE로
- 점수 랭킹: 가격(50) + 가성비(20) + 위치(40) + 계약가능(8) + 최신(7) + 특징(5)
- 위치 필터: bounding box → 텍스트 ILIKE (근본 해결)
- 오타 교정: 30개+ (오억→5억, 아빠트→아파트, ㅈㅅ→전세)

### 매칭 엔진
- match-properties.ts: SQL 우선 + 벡터 보조 + 점수 랭킹
- auto-match.ts: 카테고리/가격/좌표 체크 + 보안
- 클라이언트 파싱 제거 → 서버 통합 처리

### 코드 감사 (12건)
- CRITICAL: admin.html 문법에러, XSS, Edge Function 인증
- HIGH: 면적/방수 없는 매물 탈락 방지, 월세 필터, 카카오 키 중앙화
- MEDIUM: 임베딩 텍스트 통일, 면적 상수 통일

### 테스트 인프라
- search_test.html: 295개 키워드
- test_search_agent.py: CLI 자동 테스트
- seed_district_350.py: 25구 × 350개 데이터 생성

## 핵심 교훈

1. **텍스트 매칭 > bounding box** — 구 경계는 사각형이 아님
2. **저장 시점에 정확히 파싱** — 매칭 때마다 실시간 파싱은 낭비
3. **실제 배포 코드로 테스트** — curl로 서버에서 받아서 실행
4. **모바일은 필수 기능만** — PC CRM의 축소가 아니라 별도 설계
5. **중개사 입장에서 시뮬레이션** — "콜백", "퇴근후 전화" 같은 실무 패턴
6. **MVP 단계에서는 빠르게** — 코드 구조보다 기능 출시 속도

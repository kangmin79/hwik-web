### 3. DB 저장 4경로 일관화
파싱 → 저장 경로는 4개인데 서로 필드 저장이 제각각:
- `confirmSave` (매물 단건)
- `fiSaveAll` (매물 일괄)
- `registerClient` (손님 신규)
- `saveClientEdit` (손님 재분석)

**🚨 fiSaveAll이 id 컬럼 없어서 cards.id NOT NULL 위반 → 일괄 저장 100% 실패 상태**
**🚨 batch-parse 응답에 rawText 없어서 `property.rawText=""` 로 저장** → _doBatch/doBatch에 chunk fallback
**🚨 lat/lng를 property 내부에 저장** (top-level 컬럼이 정답) → saveEdit도 같은 버그

추가 누락:
- fiSaveAll: private_note, contact_name/phone, move_in_date, wanted_trade_type, agent_comment 전부 빠져있었음
- registerClient: property.address 3종, moveIn, management_fee/rights_money 빠짐
- saveClientEdit: tags/move_in_date/search_text_private 빠짐

DB round-trip 테스트 4/4 통과. 커밋 `27764cc`.

### 4. 손님 파싱 + 매칭 E2E
손님 매칭 기준 필드: wanted_trade_type / wanted_categories / wanted_conditions / embedding / tags / lat-lng

**손님 파싱 25케이스 테스트에서 발견:**
- 손님 `location/category` 필수가 너무 엄격 ("역세권만" 입력 시 500 에러)
- AI가 복수 카테고리 언급 시 category=null 반환 → wanted_categories[0]로 폴백 필요
- 반전세를 전세로 오분류

**🚨🚨 매칭 E2E에서 auto-match 실제 작동 검증하다 발견한 프로덕션 치명 버그 2건:**

1. **match_notifications UNIQUE 제약 부재** — auto-match의 upsert가
   `onConflict: 'agent_id,card_id,client_card_id'` 쓰는데 DB에 해당 UNIQUE가 없음.
   PostgREST가 `42P10 "no unique or exclusion constraint matching ON CONFLICT"` 400 에러.
   → **모든 매칭 알림이 저장 실패. 중개사는 매칭 알림을 한 번도 못 받았을 것**.
   마이그레이션 `20260420120000_match_notifications_unique.sql`로 복구.

2. **pgvector 문자열 → 숫자 배열 파싱 누락** — PostgREST가 vector 컬럼을
   `'[0.1,0.2,...]'` 문자열로 반환하는데 auto-match는 `target.embedding[i] * ...` 곱셈.
   문자열 인덱싱은 단일 문자 → NaN → finalScore NaN → THRESHOLD 항상 미달.
   → **벡터 유사도 기반 매칭이 구조적으로 0% 작동**.
   `parseEmb()` 추가 (Array/string 둘 다 정규화).

매칭 E2E 8/8 통과. 커밋 `066d31c`.

# 2026-04-19 아파트 단지 매칭 시스템 대개선 + ES256 인증 해결

## 한 줄 요약
중개사 매물 → 구글 유입자 danji 연결의 **생명줄 `cards.kapt_code`** 를 **2% → 91%** 정확률로 끌어올림. Supabase ES256 JWT 업그레이드로 생긴 전 Edge Function 401 이슈 해결.

## 배경 — 왜 이 작업을 했나
- 사용자 목표: 구글 유입자 → `danji.html?kapt=...` → "이 단지 휙 등록 매물" → 중개사 연결 (돈 버는 플라이휠 마디 ①)
- 생명줄: `cards.kapt_code` 정확해야 danji 페이지에 연결됨
- 현재 0 실사용자. 매물 6,583건 전부 테스트 데이터 → 깔끔히 리셋 후 재측정
- 사용자 대원칙: **"매칭 못하면 연결 금지"** (False positive = 치명)

## 사용자의 핵심 통찰 (세션 중)
1. "중개사 매물의 좌표도 중요하지?" — 플라이휠 마디의 본질 재확인
2. "임의 데이터 삭제 후 재등록 테스트" — 정확한 측정을 위한 리셋
3. "주소(도로명/지번)만 정확히 넣으면 매칭 가능?" — parse에 의존 말고 주소 기반 매칭 설계
4. "apartments DB에 도로명/지번 좌표 추가 저장 후 중개사 입력과 매칭" — 결정적 설계 아이디어
5. "구+동+단지명 + 도로명 + 지번 3가지 waterfall" — 99% 커버리지 방향
6. "정확 매칭 못 하면 제외" — 엣지 케이스 대응보다 거부가 안전

## 한 일 (완료, 순서대로)

### 1. 테스트 환경 리셋
- 백업: `backup_20260419/` (cards 6,583 / card_shares 4,934 / match_notifications 20 / client_notes 135 / memos 16)
- cards 전체 삭제 (연관 테이블 cascade 처리)

### 2. parse-property 개선
- `address` 단일 필드 → `address_road` + `address_jibun` 분리

- 단지명 괄호 보존 ("느티마을(3단지)(공무원)")

### 3. apartments 스키마 확장 + geocode 배치
- 신규 컬럼: `doro_lat`, `doro_lon`, `jibun_lat`, `jibun_lon`, `geocode_status`, `geocode_at`
- 전국 30,999건 카카오 geocode 배치 (Node.js 병렬 8 worker, 약 9분)
- 결과: ok 25,355 (81.8%) / doro_only 5,479 (17.7%) / jibun_only 97 / failed 68
- **도로명 커버리지 99.5%** / 지번 커버리지 82.1%

### 4. 신규 RPC 2개
- `match_apartment_by_coord(p_lat, p_lng, p_radius_m, p_addr_type)`: 반경 50m 내 단지 거리순
- `match_apartment_by_name_region(p_complex, p_sgg, p_umd, p_lat, p_lng)`: 이름+지역 fallback

### 5. locate-card Waterfall 3단계
- Step 1: 도로명 주소 → 카카오 geocode → `doro_lat/lon` 반경 매칭
- Step 2: 실패 시 지번 주소 → `jibun_lat/lon` 반경 매칭
- Step 3: 실패 시 `match_apartment_by_name_region` (이름+sgg+umd)
- 공식 K-apt A코드 우선 (비공식 apt-*/offi-* 중복 레코드 자동 해결)
- 이름 토큰 검증 필수 (반경 내 1개여도 이름 불일치면 거부 — False positive 방지)

### 6. ES256 JWT 이슈 해결 (미리 박힌 시한폭탄 해체)
- 증상: parse-property 401, 토큰 있는데도 Edge Function 실행 자체 안 됨
- 원인: Supabase JWT signing key가 HS256 → ES256으로 자동 업그레이드됨. Edge Function 게이트웨이의 `verify_jwt` 기본 옵션이 ES256 검증 실패
- 단서: `Auth.getToken()` 정상, `/auth/v1/user` 200 OK, 오직 Edge Function만 401
- 해결 2단계:
  1. `--no-verify-jwt` 옵션으로 재배포 (게이트웨이 검증 OFF)
  2. Edge Function 내부에서 `/auth/v1/user` fetch로 수동 검증 (모든 알고리즘 호환)
- `_shared/auth.ts` + parse-property + embed-card 모두 fetch 방식으로 교체
- `supabase/config.toml` 생성 — 8개 함수 `verify_jwt = false` 영구화

### 7. hub-new UX 개선
- 매물 등록 완료 후 `locate-card` 응답 기반 매칭 결과 토스트:
  - 성공: "✅ 헬리오시티아파트 단지로 매칭됨"
  - 실패: "⚠️ 단지 매칭 실패 — 정확한 도로명/지번 주소를 포함해주세요"
- placeholder + hint: "아파트는 도로명/지번 주소를 정확히 넣으면 단지 자동 매칭" 안내
- 연락처 UI: "🔒 나만 보는 정보" 박스 내부로 명확히 이동 (시각적 혼동 제거)
- Auth.getToken() localStorage fallback (세션 초기 로드 타이밍 이슈)

## 남은 미해결 버그

### Step 3 정규식 한글 `\b` 버그
- `[가-힣]+(?:동|읍|면|리)\b` 의 `\b`가 한글 뒤 word boundary 인식 못 함
- 현재 `umd` 추출 실패 → Step 3 전부 미작동
- 수정 (5분): `/([가-힣]+(?:동|읍|면|리))(?=[\s,.]|$)/`
- 실전 중개사 입력 패턴("강남구 대치동 은마")에서 효과 큼

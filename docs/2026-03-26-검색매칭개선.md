## 주요 작업 내역

### 1. 로컬 파서 추가 (localParseQuery)
- Claude API 없이 즉시(0ms) 검색어 파싱
- 거래유형, 카테고리, 지역(25구+70동), 가격, 단지명, 특징 인식
- 오타/한글숫자 교정: 오억→5억, 아빠트→아파트, ㅈㅅ→전세 등 30개+

### 2. SQL 직접 검색 도입
- 구조화된 조건(거래+카테고리+가격+위치) → SQL WHERE 직접 필터
- 벡터 검색은 애매한 쿼리에만 사용
- **핵심 변경**: bounding box → property.location 텍스트 필터 (근본 해결)

### 3. 점수 기반 랭킹 시스템
- 가격 근접도 (50점): 예산 이하 & 가까운 순
- 가성비 (20점): 같은 가격이면 넓은 순
- 위치 (40점): 검색 지역에 가까울수록 (가중치 4배)
- 계약가능 (8점), 최신 (7점), 특징 (5점)
- 30점 미만 컷오프: 가격 조건 있을 때만 적용

### 4. 손님 매칭 엔진 (match-properties.ts)
- SQL 직접 검색 우선 + 벡터 보조
- 오타 교정 → 조건 추출 → SQL 필터 → 점수 랭킹
- 거래유형 하드필터 (벡터 보조 후 재확인)

### 5. 자동 매칭 (auto-match.ts)
- 카테고리/가격/좌표 거리 체크 추가
- 오타 교정 통일 (30개)
- agent_id 보안 검증

### 6. 클라이언트 파싱 제거 (핵심!)
- **문제**: my_cards.html이 검색어를 미리 파싱 → 서버와 불일치 → 테스트 통과인데 실제 실패
- **해결**: 클라이언트 파싱 제거, 원본 검색어 그대로 서버에 전송
- **결과**: 테스트 = 실제 UI = 동일한 결과

### 7. 위치 필터 근본 수정
- **문제**: bounding box(좌표 사각형)는 구 경계가 불규칙해서 인접 구 포함
- **예**: 중랑구 bbox에 동대문구 이문동 포함
- **해결**: SQL에서 `property.location ILIKE '%중랑%'` 텍스트 직접 필터
- **결과**: 25개 구 전부 100% 정확

## 핵심 교훈
- **부동산 검색에서 지역은 필수** — 지역 없는 검색은 의미없음
- **구조화된 데이터 + SQL 직접 필터가 벡터보다 정확** — 벡터는 보조
- **클라이언트 파싱은 위험** — 서버에서 통합 처리가 맞음
- **테스트와 실제가 같아야** — 호출 방식이 다르면 테스트 무의미
- **bounding box < 텍스트 매칭** — 구 경계는 사각형이 아님

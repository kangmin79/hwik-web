## 2. 왜 2번이나 반복됐나 (근본 원인 분석)

1. **extract_css_js() 자체가 구조적 폭탄**: "입력 소스에서 뽑아서 타깃에 쓴다" 라는 구조인데 입력 소스가 레거시 셸로 축소된 상태를 검증하지 않음. 입력이 비어 있어도 조용히 빈 껍데기를 타깃에 써버림.
2. **주석 처리는 약한 방어**: 응급 복구 시 "쓰기 블록만 주석" 은 다음 세션에서 누가 (나 또는 사용자) "어? 이거 왜 주석이지?" 하고 되살릴 위험이 남아 있음. 사고 재발 비용이 높다면 주석이 아니라 **코드 제거**가 맞음.
3. **패턴 맹목**: Day 1 수정 직후 `css, js = ...` 튜플 선언의 대칭성을 인식했어야 했는데 "성공" 신호가 뇌를 완료 모드로 전환시킴. 특히 긴 세션에서 심화.
4. **HEAD 200 ≠ 브라우저 렌더**: 기존 `verify_seo.py` 가 HEAD/GET 만 확인하므로 "서버는 정상 응답, 브라우저엔 리치 UI 없음" 상황을 못 잡음. Playwright 기반 DOM 검증이 필수인데 **production 배포 후에만 돌렸음** — 이미 늦음.

## 6. 오늘의 배운 것 (다음 긴 세션을 위한)

1. **주석으로 막지 말 것** — 위험한 로직은 통째로 삭제. 주석은 다음 세션에서 되살림.
2. **튜플 선언 = 대칭 경고** — `a, b = ...` 를 볼 때마다 "a 처리가 있으면 b 처리도 있음" 을 의식.
3. **Pre-push > rollback** — 되돌리는 자동화보다 "나쁜 걸 애초에 안 만드는" 자동화가 훨씬 견고. 롤백 스크립트는 그 자체가 새 버그 표면.
4. **Playwright DOM 체크는 배포 후가 아니라 배포 전에** — HEAD 200 이 지나간 자리에 브라우저 버그가 숨는다.
5. **긴 세션 + 성공 신호 = 패턴 맹목 위험** — 한 군데 고쳐서 성공하면 뇌가 완료 모드로 전환. 커밋 직전 "같은 파일의 자매 블록 grep" 을 명시적 체크리스트로.

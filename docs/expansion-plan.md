## 최적화 전략

### 1. 증분 수집 (전체 재수집 X)

현재: 매일 전체 25개 구 수집 → 200 API 호출
개선: **변경 가능성 있는 것만 수집**

```
방법: trade_cache에 last_fetched 타임스탬프 추가
  → 당월 데이터만 매일 수집 (66 × 4 = 264회)
  → 전월 데이터는 3일에 1번만 (늦은 신고 반영)
  → 2개월 전 이상은 수집 안 함

결과: 264회/일 (매일) + 264회/3일 (전월) = 평균 352회/일
현재 200회 대비 1.8배 → 수집 시간 ~27분
```

### 2. 증분 집계 (전체 재집계 X)

현재: 매일 전체 단지 재집계 → 25,000개 로드+계산+저장
개선: **새 거래가 있는 단지만 재집계**

```
방법:
1. 수집 단계에서 "새 거래 있는 구" 목록 기록
2. 집계 시 해당 구의 단지만 재집계
3. 지하철/학교는 변하지 않으므로 → 최초 1회만, 이후 스킵

예상:
  당월 새 거래 있는 구: ~40개 (66개 중)
  해당 구 단지: ~15,000개
  실제 재집계: ~8,000개 (거래 있는 단지만)
  → 현재 대비 60% 수준
```

### 4. GitHub Actions 스케줄

```yaml
# 매일 새벽 1시: 당월 수집 (전 지역) + 증분 집계
- cron: '0 16 * * *'

# 3일마다 새벽 2시: 전월 수집 (늦은 신고 반영)
- cron: '0 17 1,4,7,10,13,16,19,22,25,28 * *'
```

매일 1개 워크플로우만 실행 → GitHub Actions 분 절약.

### 5. apartments 초기 시딩

인천/경기 단지 데이터 수집 방법:
1. K-apt API에서 단지 목록 가져오기 (1회성)
2. 카카오/네이버 지오코딩으로 좌표 추가
3. sync_pyeongs.py로 공급면적 수집

```bash
# 인천 단지 시딩
python seed_apartments.py --region incheon

# 경기 단지 시딩 (주요 도시만 먼저)
python seed_apartments.py --region gyeonggi --cities 수원,성남,용인,고양,화성
```

### 6. DB 용량 관리

```
trade_cache 최적화:
  현재: 구+유형+월별로 JSON 배열 저장 (큰 row)
  개선: 3년 이전 데이터 자동 삭제 (cron job)

  DELETE FROM trade_cache
  WHERE year_month < to_char(now() - interval '36 months', 'YYYYMM');

예상 용량:
  apartments: 25,000 × 2KB = 50MB
  trade_cache: 15,000 × 20KB = 300MB
  danji_pages: 20,000 × 5KB = 100MB
  총: ~450MB (500MB 무료 한도 근접!)

→ Supabase Pro ($25/월) 전환 시점: 경기 확장 완료 후
```

### 8. SEO 페이지는 코드 변경 없음

gu.html, dong.html, ranking.html은 **DB 쿼리 기반**이라:
- 인천/경기 데이터가 danji_pages에 들어가면
- 자동으로 페이지 생성됨
- sitemap도 자동 확장

유일하게 필요한 것: gu.html의 GU_LIST에 인천/경기 추가.

## 1. 문제

직방, 다방, 네이버부동산 — 매물이 보이는 순서는 매물 품질이 아니라 중개사가 낸 광고비 순서다. 월 90~200만원을 내야 상위에 뜬다. 못 내면 묻힌다.

중개사 75,000명이 이 구조 안에서 매달 수백만원을 태우고 있다.

손님 입장에서도 문제다. 상위 매물이 좋은 매물이 아니니까 "클릭 → 상세 → 뒤로" 수십 번 반복한다. KISO 부동산매물클린관리센터 통계에 따르면 2024년 허위매물 신고 29,812건 중 21,553건이 실제 허위로 판정됐다. 2026년 들어서는 전년 대비 47% 증가했다. (출처: 한국인터넷자율정책기구 거짓매물 처리현황, 아주경제 2026.03.31)

그리고 매물 등록 자체가 고통이다. 사진 10장, 30개 항목 수동 입력, 매물 하나에 20~30분. 바쁜 중개사는 결국 카톡 단체방에 "전세 3억 강남 30평 5층" 한 줄 보내는 게 현실이다.

### 현장 목소리

서비스 기획 단계에서 수도권 중개사무소 5곳을 방문해서 물어봤다.

- **A 대표 (마포구, 경력 12년)**: "직방 광고비만 월 150만원. 안 내면 매물이 아예 안 보여요. 그런데 내도 효과가 있는지 모르겠어요."
- **B 실장 (강남구, 경력 8년)**: "매물 하나 올리는 데 20분. 하루에 5개 올리면 그게 오전이에요. 차라리 카톡으로 보내는 게 빨라요."
- **C 대표 (분당, 경력 15년)**: "손님이 허위매물에 지쳐서 전화를 안 해요. 진짜 매물인데도 의심부터 합니다."
- **공통 반응**: 광고비 부담이 가장 크고, 매물 등록이 번거롭고, 손님과의 신뢰가 깨져 있다.

5명 중 5명이 "광고비 없이 매물이 노출되면 당장 쓰겠다"고 했다.

## 2. 솔루션

휙은 중개사가 텍스트 한 줄 치면 AI가 매물 카드를 만들어주고, 손님 조건에 맞는 매물을 자동으로 매칭해주는 서비스다.

```
중개사 입력: "래미안 84평 전세 7억 15층 올수리 역세권 즉시입주"

→ AI가 파싱:
  단지: 래미안 | 면적: 84㎡ | 전세 7억 | 15층 | 올수리, 역세권, 즉시입주

→ 매물 카드 생성 → 카톡으로 손님에게 전송
→ 손님은 스와이프로 매물 비교
→ 동시에 AI가 조건 맞는 손님에게 자동 알림
```

광고비 0원. 등록 30초.

| | 직방/다방 | 휙 |
|---|---|---|
| 매물 등록 | 30분 수동 | 30초 AI |
| 노출 기준 | 광고비 | AI 매칭 |
| 손님 UX | 목록→상세→뒤로 반복 | 스와이프 |
| 데이터 | 중개사 자가 입력 | 국토부 공식 |
| 비용 | 월 90~200만 | 0원 |

## 3. 기술 스펙

아이디어가 아니라 이미 돌아가고 있는 서비스다. 라이브 URL: **hwik.kr**

### 3-1. 시스템 구성

```
프론트엔드:  HTML/CSS/JS (빌드 도구 없이 단일 파일, GitHub Pages 배포)
백엔드:     Supabase (PostgreSQL + pgvector + Edge Functions + Auth + Storage)
AI 모델:    Anthropic Claude Haiku 4.5 (매물 파싱) + OpenAI text-embedding-3-small (벡터)
인증:       카카오 OAuth
지도:       카카오맵 SDK
자동화:     GitHub Actions (cron, 매일 01:00 KST)
도메인:     hwik.kr (GitHub Pages + CNAME)
```

### 3-2. AI 매물 파싱 (Edge Function: parse-property)

중개사가 자유 텍스트를 입력하면 Claude Haiku가 구조화 데이터로 변환한다.

- **입력**: `"래미안 84평 전세 7억 15층 올수리 역세권"`
- **처리**: Claude API → JSON 추출 (type, price, location, complex, area, floor, features)
- **후처리**: 면적 ㎡↔평 변환, 가격 파싱(억/천/만), 국토부 API로 해당 단지 실거래가 조회
- **출력**: 구조화된 매물 데이터 + 벡터 임베딩 + AI 자동 코멘트
- **응답 시간**: 1~2초

### 3-3. 벡터 검색 (Edge Function: search-property)

"강남 전세 5억 이하 신축"처럼 자연어로 검색하면 의미 기반으로 결과를 돌려준다.

- **임베딩 모델**: OpenAI text-embedding-3-small (1,536차원)
- **저장**: PostgreSQL pgvector 확장 (cards.embedding 컬럼)
- **검색 방식**: 하이브리드 — SQL 하드필터(거래유형/가격/지역) + 벡터 코사인 유사도
- **가격 파싱**: "5억 이하", "3억~5억", "보증금 1000에 월세 50" 등 자연어 가격 조건 처리
- **필드 가중치**: 13개 필드별 점수 (거래유형 10, 단지명 8, 위치 8, 면적 6, 층수 5...)
- **검색 모드**: 내 매물 / 공유 매물 / 손님 카드 3개 병렬 실행
- **응답 시간**: 200ms 이내

### 3-4. 자동 매칭 (Edge Function: auto-match)

매물이 등록되면 손님 카드와 자동으로 매칭한다. 125점 만점 알고리즘:

```
거래유형 일치:   40점   (불일치 = 0점, 즉시 탈락)
가격 범위:      30점   (기준가 5% 초과당 -2점)
위치 근접도:    20점   (같은 동 20, 1km 15, 3km 10, 5km 5)
매물 유형:      15점   (아파트/오피스텔/원룸 등)
태그 매칭:      20점   (필수 태그 10, 선호 태그 5)
제외 태그:     -50점   (즉시 탈락)
임베딩 보너스:   +5점   (태그가 부족할 때 의미 유사도 보정)
```

- **거리 계산**: Haversine 공식 (위경도 → 직선거리)
- **거리 보너스**: 500m +0.25, 1km +0.20, 2km +0.12, 5km +0.05
- **임베딩**: 태그 매칭 점수가 낮을 때만 벡터 유사도로 보정 (과적합 방지)
- **처리 속도**: 손님 100명 기준 200ms

### 3-5. 실거래가 자동 수집 파이프라인

국토부 실거래가 공개 API에서 매일 새벽 데이터를 수집해서 DB에 넣고, 정적 페이지를 자동 생성한다.

```
GitHub Actions (매일 01:00 KST)
  ├─ Job 1: 서울 25개구       ─┐
  ├─ Job 2: 인천 10개구/군     ├─ 3개 병렬 실행 (각 90분 타임아웃)
  └─ Job 3: 경기 39개 시/군/구 ─┘
       │
       ▼
  sync_trades.py (Python)
    ├─ 국토부 API 호출: 아파트 매매/전세/월세 + 오피스텔
    ├─ 거래 → trades 테이블 upsert
    ├─ 단지별 집계 → danji_pages 테이블 (최근거래, 역대최고가, 전세가율, 가격이력)
    ├─ 주변 시설 매칭: 지하철역(Haversine 1km), 학교(1km), 주변단지(2km)
    └─ 이메일 보고서 발송
       │
       ▼
  build_danji_pages.py → 11,165개 정적 HTML (단지별)
  build_dong_pages.py → 595개 정적 HTML (동별)
  sitemap.xml → 11,841 URL
       │
       ▼
  git push → GitHub Pages 자동 배포
```

- **데이터 소스**: 국토부 RTMSDataSvc API (아파트/오피스텔/전월세)
- **수집 범위**: 서울 + 인천 + 경기 (3개 광역, 74개 시군구)
- **이력**: 최근 36개월 (초기 적재) → 이후 매일 최근 2개월 갱신
- **매칭 로직**: 국토부 거래의 단지명 → apartments 테이블의 kapt_code (키워드 매칭)
- **집계 항목**: 평형별 최근거래, 역대최고가, 전세가율, 가격추이, 거래량

### 3-6. SEO 정적 페이지

각 단지/동 페이지는 사전 렌더링된 HTML이다. JS가 로드되기 전에도 Google이 크롤링할 수 있다.

- **단지 페이지 (11,165개)**: 실거래가, 1년 전 비교, 주변 지하철/학교, FAQ 7개, 주변 단지 링크
- **동 페이지 (595개)**: 동네 리포트(단지 수, 최고/최저가, 최다 세대), 비교 문장, FAQ 7개
- **구조화 데이터**: JSON-LD (Residence + BreadcrumbList + FAQPage)
- **Google Search Console**: 11,841 URL 등록, 매일 sitemap 갱신

## 5. 왜 내가 하는가

네이버 마케팅을 15년 했다. 키워드 광고, 블로그 마케팅, 검색 최적화 — 광고비로 노출 순위를 사는 구조를 안에서 직접 운영해왔다.

그래서 이 구조의 문제를 안다. 광고비를 많이 내는 쪽이 이기는 시스템에서는 품질이 아니라 예산이 경쟁력이다. 부동산도 똑같은 구조다. 15년 동안 그 안에 있었으니까 어디가 비효율이고 어디를 기술로 대체할 수 있는지 보인다.

개발 전공은 아니다. Claude Code(AI 코딩 도구)로 대화하면서 만들었다. 5주 동안 2,000번 넘게 코드를 고쳤다. 마케팅 경력이 기술과 만나면 뭘 할 수 있는지 — 이 서비스가 그 답이다.

정리하면:
- **15년 마케팅 경력** → 광고비 구조의 문제를 안에서 체감한 사람
- **검색/SEO 전문성** → 11,841 페이지를 Google에 최적화하는 건 이 경력이 있어서 가능했다
- **AI 도구 활용력** → 전공 없이 5주 만에 실제 서비스를 만들어 운영 중

## 6. 비즈니스 모델

지금은 무료다. 중개사 네트워크부터 만든다.

프리미엄 구독(월 5~20만원)으로 전환한다. 중개사가 직방에 내는 광고비(월 90~200만원)의 10분의 1이다.

직방/다방은 광고비가 매출이라 AI 매칭을 도입하면 자기 수익 모델이 무너진다. 구조적으로 못 한다.

솔루션: AI가 매물을 파싱하고, 조건에 맞는 손님과 자동 매칭. 광고비 0원.
현재: 5주 만에 만들어서 이미 돌아가고 있다. 13,418 단지, 11,841 페이지, 매일 자동 갱신.
